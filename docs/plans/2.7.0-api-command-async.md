# Plan: Async `/api/command` for 2.7.0

Target: **2.7.0**.

## TL;DR — recommended approach

**Hybrid of options 1+2: in-process short-wait queue with optional
fast-fail.**

The HTTP handler enqueues the BAPI command on a FreeRTOS queue
drained by the existing BLE task. It then waits a *short* deadline
(default 800 ms, `?wait=ms` tunable up to ~2500 ms) on a per-request
response notifier:

- **Response arrives in time** → return `200 + body`, byte-for-byte
  the same JSON shape today's callers get. Most happy-path requests
  (BAPI completes in 200-500 ms on healthy MAX).
- **Deadline expires** → return `202 Accepted + {id, status:"pending"}`.
  The BLE task continues; when the response lands it goes to a fixed
  MQTT topic (`…/response/<met>`) AND a WebSocket `cmdresp` channel.
- **Poll path** → `GET /api/command_status?id=N` returns the result if
  available, or 410 Gone if it expired out of the RAM map (FIFO, 4
  entries).

### Why not pure 202 (option 3)

- The web UI and curl scripts both parse `d.r` / `d.error` from the
  response body. Audit found 14 call sites in `wb_web.cpp` that
  read the response. Pure 202 would silently break all of them.
- ~98 % of requests complete in <800 ms on healthy hardware. The
  async path is only the fallback, not the default experience.

### Why not pure long-poll (strict option 2)

- The maintainer's web UI uses `AbortSignal.timeout(15000)` for slow
  commands. Pure long-poll wins nothing over today's 5 s blocking
  sync — just delays the failure differently.

### Why not pure WebSocket push (option 4)

- HA and curl don't speak WS. Drops two important non-HA caller
  classes.

### Why not fire-and-forget (option 5)

- Kills response-bearing endpoints. Breaks back-compat for ~14 UI
  callsites + any user curl scripts.

## Architecture

### New BLE-task queue (`BleReq`)

```c
struct BleReq {
    uint32_t  reqId;
    char      met[16];
    char      par[192];     // BAPI payloads are small; truncate
    uint8_t   replyMode;    // 0 = fire-and-forget, 1 = wake-waiter, 2 = mqtt-publish
    TaskHandle_t waiter;    // xTaskNotify target, NULL if fire-and-forget
    uint32_t  enqueuedAt;   // for timeout + drop-old logic
};
```

Backed by FreeRTOS `QueueHandle_t`. Depth **6** (matches the existing
`MAX_API_CMD_INFLIGHT=2` × ~3-deep MQTT burst headroom). Queue full
returns `kEnqueueBusy` → handler responds `503 busy` (same as today).

### Response-side correlation

The BAPI `id` field correlates response to request. `ResponseParser`
already extracts it (`responseId()`). On notification:
1. Copy response JSON to a RAM map keyed by `reqId` (cap 4, FIFO).
2. If originating `BleReq` had a `waiter`, `xTaskNotify(waiter, reqId,
   eSetValueWithOverwrite)`.
3. If `replyMode == mqtt-publish`, push to a "publish-from-main"
   queue (PubSubClient is NOT thread-safe — see existing
   `_storeCache` discipline).

### Response-publishing back-channel (BLE task → main task)

**Critical constraint:** PubSubClient cannot be called from the BLE
task. Pattern (mirrors `_storeCache`):

- Add `String _pendingMqttPub[4]` + matching met[] + head/tail
  indices, all under `_cacheMutex`.
- New `drainPendingResponsePub(String& met, String& json)` called
  from main task in the existing `publishCached*IfNew` block.
- `wbws::broadcast("cmdresp", ...)` added to the drain loop.

## Backward compat

**Default = transparent async.** Same URL, same JSON shape on success.
Only observable change: occasional 202 + `{id, status:"pending"}`
instead of waiting up to 5 s. This is *better* than the current
behavior even for naïve callers (they used to hang then get
`{error:"timeout"}`).

**Tweak to `wb_web.cpp:337` JS retry policy:** `isCmd()` currently
treats non-2xx as retry-with-backoff. 202 should be success. Both
dashboard buttons (`C(a)` line 881, `setCurrent` line 883) already do
"toast then re-poll `/api/charger`" — they don't depend on the
response body, so they keep working unchanged.

**Explicit opt-ins via query string:**
- `?wait=0` — pure async, returns 202 immediately. For HA scripts.
- `?wait=2500` — extend short-wait up to the hard cap. For UI commands
  that need the answer inline.
- `?sync=1` — fall back to current 5 s blocking behavior. Documented
  as deprecated; kept for two releases as escape hatch. Concurrent
  `?sync=1` capped at 1 (not 2) so a stuck sync wedge can't double up.

## MQTT `_handleCommand` parity

**Yes — it's actually the bigger win.** Every HA toggle goes
through MQTT subscribe → `_mqttCallback` → `_handleCommand` (running
inside `_client->loop()`, on the main task). The 5 s sync
`sendCommand` blocks main loop the same way `/api/command` does.

All 14 `sendCommand` calls in `_handleCommand` discard the return
value. Switch them all to `enqueueRequest(fire-and-forget)`. The raw
`/bapi` topic at line 373 switches to `mqtt-publish`. Net: HA's
command path no longer blocks main loop at all.

The `requestSettingsRepoll()` call at line 508 still fires correctly
— it gets the new state back to HA, async, ~150-500 ms after the
command lands. Bump its default delay from 500 ms → 750 ms when
called from `_handleCommand` post-refactor to give the queued write
extra head-start.

## Rate-limiter (`_tbTokens`) interaction

**Consume the token on enqueue** (current behavior, just relocated).
Rationale:
- The bucket gates *demand* on the BLE radio — preventing flood
  faster than BAPI can drain. The radio cost is incurred whether or
  not HTTP handler waits.
- Consuming on response would let a stuck curl loop fill the queue
  with 6 requests + another waiting cap-reject = unbounded backlog.
- The HA `tokens` metric stays semantically correct: "tokens you
  have left before next request is rate-limited."

`MAX_API_CMD_INFLIGHT` gets *removed* — queue-depth cap (6)
supersedes it. The 503 BUSY path stays for queue-full. (Or keep
gated to `?sync=1` only.)

## Implementation order

1. **[S] Add `BleReq` struct + queue handle to `WallboxBLE`.** Allocate
   in `begin()`. No behavior change. Verify link + boot.
2. **[M] Implement `enqueueRequest` + drain in `loop()`.** Drain to
   existing `sendCommand` path internally. End state: enqueued
   requests work, but everything still sync from caller POV.
3. **[M] Add response RAM-map + reqId correlation in `_notifyCb`.**
   No caller-visible change. Long soak with existing sync path to
   confirm `responseId()` reliability.
4. **[M] Refactor `sendCommand` to thin wrapper around
   `enqueueRequest + xTaskNotifyWait`.** Single code path. Biggest
   win for confidence — any latent regression shows in *every*
   existing caller.
5. **[M] Add pending-MQTT-pub queue + main-task drain.** Wire
   `wallboxMQTT.publishResponse` for `mqtt-publish` replies. Test via
   raw `/bapi` MQTT topic.
6. **[M] Add `handleApiCommand` async path.** Short-wait + 202
   fallback, `?wait`, `?sync=1`, `?wait=0`. Keep `_apiCmdInflight`
   gating only on `?sync=1`. Tweak `isCmd()` JS.
7. **[S] Add `handleApiCommandStatus` route.**
8. **[M] Convert `_handleCommand` to `enqueueRequest(fire-and-forget)`.**
   Switch `/bapi` raw to `mqtt-publish`.

10. **[S] CHANGELOG + docs.**

## Test plan (live on maintainer's MAX)

Baseline first: capture `loop_max_ms` for (idle), (1 HA toggle), (5
HA toggles in 2 s), (20× curl loop with 0.1 s sleep).

### Smoke (after step 4)

- Web UI dashboard renders. No regression in existing flows.
- `/api/command?action=bapi&met=g_tzn` returns 200 + body, same JSON.
- `loop_max_ms` over 5 min soak: unchanged from baseline.

### Async happy path (after step 6)

- HA "Start charging": `loop_max_ms` < ~50 ms (vs baseline 500-800 ms).
- 10× curl loop tight on `/api/command?met=g_tzn`: all return 200
  within 800 ms; `loop_max_ms` stays sub-100 ms.
- `r_ses` (slow): first hit returns 202 + `{id,status:pending}` after
  800 ms; `/api/command_status?id=N` polls succeed; body matches
  `?sync=1`.

### Async stress (after step 8)

- 30 HA toggles in 5 s via automation: gateway doesn't wedge.
  `loop_max_ms` < ~200 ms. First ~6 hit the radio; queue-full
  returns 503 BUSY (rate-limit). HA "Start charging" still works.
- curl tight loop in background, 30 min: heap stable, no leak in
  `_pendingMqttPub` ring, no reboot.

### Backward compat

- Dashboard / settings / sessions / BAPI explorer — every response-body
  parser still gets data within its `AbortSignal.timeout`.
- Simulate scripted curl with `?wait=0` → confirm 202 → poll
  `/api/command_status` returns same shape.

### MQTT raw `/bapi` topic

- Publish `{"met":"r_dat","par":null}` → observe `…/response/r_dat`
  retained within ~500 ms, identical shape to today.

## Risks

| Risk | Severity | Mitigation |
|---|---|---|
| HA expecting sync responses | LOW. HA MQTT path doesn't read response. UI buttons don't read `r`. | Per-callsite audit at step 6. Slow UI commands use `?wait=2500`. |
| Curl scripts hardcoded 200 | LOW-MED. 202 is 2xx, `curl --fail` accepts it. Only breaks scripts that assert `status_code == 200`. | `?sync=1` escape hatch; CHANGELOG headlines. |
| Request/response ID collision | LOW. Monotonic int, 248 days to wrap at 100 req/s. | FIFO cap 4; expired ID returns 410 Gone. |
| `tokens` metric semantics shift | LOW. Same value, different timing. | CHANGELOG. |
| `?sync=1` still wedges | LOW. By design. Drop in a release or two. | Caller opted in. |
| BLE task starvation if response-pub queue overflows | MED. If MQTT wedged AND every command publishes a response. | Drain at ~50 Hz; cap 4 means BLE task drops oldest pub with log line; rare in practice (only `/bapi` raw and `_handleCommand` paths). |
| 800 ms default too aggressive | LOW. More fallthroughs to 202 on degraded BLE; UX is "instant pending"; request still completes. | Make tunable via `/config` if users complain. |
| `requestSettingsRepoll` timing | LOW. Today it's called after sync `sendCommand` returns. With async, repoll requested before write lands. | Existing 500 ms default covers this; bump to 750 ms when called from `_handleCommand` post-refactor. |
| Reentry tripwire pressure | LOW. Async actually reduces it — `xTaskNotifyWait` doesn't call `onYield`. | Verify `max_reentry` MQTT field stays at 1. |
