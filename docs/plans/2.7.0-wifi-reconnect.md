# Plan: Move WiFi.reconnect() Off The Main Loop

Target: **2.7.0**.

## Recommended approach

**Event-driven (Arduino-ESP32 `WiFi.onEvent`) primary + exponential
backoff defer policy.** Event handler sets flags only; `wb_net::tick()`
on the main task drives the explicit `WiFi.reconnect()` call **only**
when:
- the driver's own auto-reconnect has clearly given up (≥60 s since
  last `STA_GOT_IP`), AND
- the backoff window has elapsed (1, 2, 5, 15, 60 s ladder).

Most of the time the driver auto-reconnects on its own and we never
explicitly call `WiFi.reconnect()`; the main loop never blocks.

### Why not the other options

- **Dedicated FreeRTOS task** (option 1): no per-tick work for it to
  own. Would spend 99.99 % of its life calling `WiFi.status()` and
  sleeping. Wasteful.
- **Bounded-timeout wrap** (option 3): Arduino-ESP32 doesn't expose a
  timeout knob on `WiFi.reconnect()`. The work that blocks isn't
  the call — it's the driver's downstream scan/auth/DHCP work.
- **Defer-only** (option 4): reduces cadence (every 30 s → every
  60 s) but doesn't eliminate the block. Combine with option 2,
  don't use alone.

## Synchronization

- WiFi event callback runs on the **WiFi driver's event task**
  (separate FreeRTOS task internal to esp_wifi, ~core 0).
- Main loop on core 1 (Arduino default).
- Shared state: `_connected` bool (atomic / volatile — single-word on
  Xtensa); `_lastConnectedAt` / `_lastDisconnectedAt` (uint32 millis);
  backoff timer state.
- Event handler **only sets flags** — all real work (mDNS refresh,
  `wb_diag::reportDisconnect/Reconnect`, `WiFi.reconnect()` call)
  deferred to `wb_net::tick()` on main. Same pattern BLE uses for
  its `pause()` flag.

## Risks

- **Deauth loop if backoff misconfigured.** Mitigation: clear
  `_wantsReconnect` + reset `_backoffMs` on `GOT_IP`; assert
  `!_connected` before any `WiFi.reconnect()` call.
- **mDNS re-init flakiness.** Defer to main task. If unstable, fall
  back to "mDNS re-registers on next reboot" — degraded but not broken.
- **WiFi event callback stack overflow** if we do heavy work there.
  Mitigation: handlers only set flags + timestamps. All work
  deferred to `tick()`.
- **`_connected` cache lag vs live `WiFi.status()`.** Main-loop hot
  path uses cached; display sites (wb_health, wb_web /info) keep
  reading live. Display-only race; no correctness impact.
- **`WiFi.reconnect()` can still block several seconds when it does
  fire.** This refactor reduces cadence (60 s → on-demand) and
  ensures the grace gate covers GOT_IP recovery. The "permanently
  dead AP" case will still register some main-loop blocks — but at
  60 s cadence, not 30 s.
- **HA discovery storm on reconnect.** 2.6.0's one-per-tick state
  machine already bounds this; no new mitigation. Mention in
  CHANGELOG so the two fixes read as related.

## Reference implementation pattern

Already in tree: `src/wb_ble.cpp:42-77` shows the BLE-task pattern
(`xTaskCreatePinnedToCore`, `vTaskDelay` tick, mutex-protected shared
state, `pause()` defer-via-flag). The WiFi refactor uses the same
defer-via-flag discipline but with the framework's event task instead
of our own.
