# Plan: Table-driven HA discovery refactor

Target: **3.0.0** (task #77).

## What this is

`WallboxMQTT::tickDiscovery()` in `src/wb_mqtt.cpp` is a 57-slot
`switch (_discoveryIndex)` that fires one HA MQTT discovery payload
per main-loop tick. Each case is a `publishDiscovery*(...)` helper
call with mostly-fixed arguments — entity name, object id, icon,
state topic, value template, unit, etc.

Goal: replace the switch with a table of `DiscoveryEntry` structs
indexed by `_discoveryIndex`. No protocol or wire-format change —
HA sees the exact same retained payloads. Smaller hot-path code,
easier to add entities (one row instead of a new case + bumping
`kDiscoveryCount`), single source-of-truth for "what entities does
the gateway publish?"

Same alternative was on the table for 2.6.0; the switch approach
was picked back then because transcribing the entity attributes
into a table all at once was high-risk during a release. Now that
the per-entity wire format is stable across 2.6.x and 2.7.0 with
no field churn, it's a safer time to do the cleanup.

## Current shape (post-2.7.0)

57 case slots in the switch, of which:

| Type | Count | Helper signature |
|---|---|---|
| `publishDiscoveryEntity` (sensor, binary_sensor) | 46 | 10+ args with many optional |
| `publishDiscoveryNumber` (number) | 3 | min/max/step + mode |
| `publishDiscoverySwitch` (switch) | 5 | on/off payloads |
| `publishDiscoverySelect` (select) | 3 | options pointer + count |
| `publishDiscoveryButton` (button) | 1 | minimal |
| **No-op slot** (case 9 — kept to avoid renumbering after a 2.6.0 drop) | 1 | — |
| **Direct JsonDocument build** (case 13 — `car_connected`, has extra availability handling) | 1 | inline |

Topic references come from `_discTopics`, a struct populated once per
discovery arm with the 6 cached topic strings:
`sTopic` / `rTopic` / `gTopic` / `mTopic` / `nTopic` / `setTopic`
plus 11 command-prefix-derived topic strings
(`cmdCurrent`, `cmdCharging`, …).

## Proposed shape

```cpp
namespace wb_mqtt {

enum class EntityKind : uint8_t {
    SENSOR        = 0,  // publishDiscoveryEntity, component="sensor"
    BINARY_SENSOR = 1,  // publishDiscoveryEntity, component="binary_sensor"
    NUMBER        = 2,  // publishDiscoveryNumber
    SWITCH        = 3,  // publishDiscoverySwitch
    SELECT        = 4,  // publishDiscoverySelect
    BUTTON        = 5,  // publishDiscoveryButton
    CUSTOM        = 6,  // inline JSON build (case 13 today)
    NOOP          = 7,  // reserved-slot placeholder
};

// Which cached topic field to use. Table holds an enum instead of a
// raw pointer because the topic strings live in WallboxMQTT and the
// table must be a static constexpr (no captures).
enum class TopicSlot : uint8_t {
    NONE = 0,
    STATUS,        // r_dat (sTopic)
    REALTIME,      // r_sta (rTopic)
    GATEWAY,       // response/gateway (gTopic)
    METER,         // response/meter (mTopic)
    NOTIFS,        // response/notifications (nTopic)
    SETTINGS,      // wallbox/settings (setTopic)
    // Command-prefix topics — for the controllable entities below
    CMD_CURRENT,
    CMD_CHARGING,
    CMD_LOCK,
    CMD_REBOOT,
    CMD_AUTOLOCK_EN,
    CMD_AUTOLOCK_TIME,
    CMD_ECO_MODE,
    CMD_ECO_POWER,
    CMD_POWER_SHARE,
    CMD_PHASE_SWITCH,
    CMD_HALO,
    CMD_TIMEZONE,
};

// One row per HA discovery entity. Many fields are optional (nullptr
// or zero) depending on EntityKind. Anonymous union avoids carrying
// "min/max/step" payload in rows that don't need it; kind tells the
// dispatcher which arm to read.
struct DiscoveryEntry {
    EntityKind   kind;
    const char*  objectId;       // "charging_power"
    const char*  name;           // "Charging Power"
    const char*  icon;           // "mdi:flash"
    TopicSlot    stateTopic;     // primary state topic
    const char*  valueTemplate;  // Jinja string

    // Sensor extras (optional, nullptr ignored)
    const char*  unit;
    const char*  deviceClass;
    const char*  stateClass;     // "measurement" | "total_increasing" | nullptr
    const char*  category;       // "diagnostic" | nullptr

    // Control extras (used when kind is NUMBER / SWITCH / SELECT / BUTTON)
    TopicSlot    commandTopic;       // CMD_*
    int          numMin, numMax;     // NUMBER
    int          numStep;            // NUMBER
    const char*  numMode;            // NUMBER, "box" | nullptr (slider default)
    const char*  switchOnPayload;    // SWITCH
    const char*  switchOffPayload;   // SWITCH
    const char* const* selectOptions; // SELECT (kEcoOptions / kHaloOptions / kTzOptions)
    uint8_t      selectCount;        // SELECT
};

static constexpr DiscoveryEntry kEntries[] = {
    // case 0 — Charging Power
    { EntityKind::SENSOR, "charging_power", "Charging Power", "mdi:flash",
      TopicSlot::STATUS, "{{ value_json.r.cp | round(2) }}",
      "kW", "power", "measurement", nullptr,
      TopicSlot::NONE, 0,0,0, nullptr,
      nullptr, nullptr, nullptr, 0 },
    // ... 56 more rows ...
};
static constexpr size_t kEntryCount = sizeof(kEntries) / sizeof(kEntries[0]);

}  // namespace wb_mqtt
```

`tickDiscovery()` becomes:

```cpp
void WallboxMQTT::tickDiscovery() {
    if (_discoveryIndex == SIZE_MAX) return;
    if (!isConnected()) return;
    if (_discoveryIndex >= kEntryCount) {
        Log.printf("[MQTT] HA discovery published (%u entities)\n",
                   (unsigned)kEntryCount);
        _discoveryIndex = SIZE_MAX;
        return;
    }
    const auto& e = kEntries[_discoveryIndex];
    const char* st  = _resolveTopic(e.stateTopic);
    const char* cmd = _resolveTopic(e.commandTopic);
    switch (e.kind) {
        case EntityKind::SENSOR:
            publishDiscoveryEntity(*_client, "sensor", e.objectId, e.name,
                e.icon, st, e.valueTemplate, e.unit, e.deviceClass,
                e.stateClass, e.category);
            break;
        case EntityKind::BINARY_SENSOR:
            publishDiscoveryEntity(*_client, "binary_sensor", e.objectId,
                e.name, e.icon, st, e.valueTemplate, e.unit, e.deviceClass,
                e.stateClass, e.category);
            break;
        case EntityKind::NUMBER:
            publishDiscoveryNumber(*_client, e.objectId, e.name, e.icon,
                cmd, st, e.valueTemplate, e.numMin, e.numMax, e.numStep,
                e.unit, e.numMode);
            break;
        case EntityKind::SWITCH:
            publishDiscoverySwitch(*_client, e.objectId, e.name, e.icon,
                cmd, st, e.valueTemplate, e.switchOnPayload,
                e.switchOffPayload);
            break;
        case EntityKind::SELECT:
            publishDiscoverySelect(*_client, e.objectId, e.name, e.icon,
                cmd, st, e.valueTemplate, e.selectOptions, e.selectCount);
            break;
        case EntityKind::BUTTON:
            publishDiscoveryButton(*_client, e.objectId, e.name, e.icon,
                cmd);
            break;
        case EntityKind::CUSTOM:
            _publishCustomEntity(_discoveryIndex);  // dispatch to small switch
            break;
        case EntityKind::NOOP:
            break;
    }
    _discoveryIndex++;
}
```

Topic resolver:

```cpp
const char* WallboxMQTT::_resolveTopic(TopicSlot slot) const {
    switch (slot) {
        case TopicSlot::NONE:             return nullptr;
        case TopicSlot::STATUS:           return _discTopics.sTopic.c_str();
        case TopicSlot::REALTIME:         return _discTopics.rTopic.c_str();
        case TopicSlot::GATEWAY:          return _discTopics.gTopic.c_str();
        case TopicSlot::METER:            return _discTopics.mTopic.c_str();
        case TopicSlot::NOTIFS:           return _discTopics.nTopic.c_str();
        case TopicSlot::SETTINGS:         return _discTopics.setTopic.c_str();
        case TopicSlot::CMD_CURRENT:      return _discTopics.cmdCurrent.c_str();
        case TopicSlot::CMD_CHARGING:     return _discTopics.cmdCharging.c_str();
        case TopicSlot::CMD_LOCK:         return _discTopics.cmdLock.c_str();
        case TopicSlot::CMD_REBOOT:       return _discTopics.cmdReboot.c_str();
        case TopicSlot::CMD_AUTOLOCK_EN:  return _discTopics.cmdAutolockEnable.c_str();
        case TopicSlot::CMD_AUTOLOCK_TIME:return _discTopics.cmdAutolockTime.c_str();
        case TopicSlot::CMD_ECO_MODE:     return _discTopics.cmdEcoMode.c_str();
        case TopicSlot::CMD_ECO_POWER:    return _discTopics.cmdEcoPower.c_str();
        case TopicSlot::CMD_POWER_SHARE:  return _discTopics.cmdPowerShare.c_str();
        case TopicSlot::CMD_PHASE_SWITCH: return _discTopics.cmdPhaseSwitch.c_str();
        case TopicSlot::CMD_HALO:         return _discTopics.cmdHalo.c_str();
        case TopicSlot::CMD_TIMEZONE:     return _discTopics.cmdTimezone.c_str();
    }
    return nullptr;
}
```

## Migration strategy

The hard part is **transcribing 55 rows correctly**. The single
biggest risk is silently swapping the order of two adjacent
optional arguments (e.g. `unit` and `deviceClass`) — HA still
accepts the payload but the entity surfaces wrong in the device
card.

Five-step migration that minimises that risk:

1. **Step A — scaffolding only.** Add the `EntityKind`, `TopicSlot`
   enums, the `DiscoveryEntry` struct, the empty `kEntries[]` array
   (with `kEntryCount = 0`), and `_resolveTopic()` helper. Build,
   no functional change.
2. **Step B — wire the dispatch.** Add the new `tickDiscovery()`
   body alongside the existing switch. Pick by a build flag
   `WB_DISCOVERY_TABLE_DRIVEN` (default off). Build with flag
   off — old behaviour. Build with flag on — discovery does
   nothing because the table is empty.
3. **Step C — transcribe entries in groups.** Convert cases in
   logical batches:
   - Group 1 (cases 0-8, 10-12): r_dat / r_sta sensors  
   - Group 2 (cases 14-17): main controls (current, charging,
     lock, reboot)  
   - Group 3 (cases 18-22): meter + BLE rssi  
   - Group 4 (cases 23-32): notifications + settings entities  
   - Group 5 (cases 33-45): charger details from gTopic  
   - Group 6 (cases 46-56): diagnostic-category entities  
   - Special: cases 9 (NOOP), 13 (CUSTOM — keep as a tiny inner
     switch in `_publishCustomEntity`)
   
   After EACH group: rebuild with the flag on, OTA, restart HA's
   MQTT integration, and verify ONLY entities in that group appear
   in the device card. Catch transcription errors group-by-group
   instead of all-at-once.
4. **Step D — flip the default.** When all 57 slots are
   transcribed and verified, change the default of
   `WB_DISCOVERY_TABLE_DRIVEN` to ON. Keep both paths in source
   for one release in case a regression surfaces.
5. **Step E — remove the legacy switch.** Next release after that,
   delete the old switch and the build flag. Single source of truth.

## What this does NOT change

- The wire format of any discovery payload. HA sees identical
  retained messages.
- The `wallbox/<sn>/...` topic structure.
- `sendDiscovery()` — it still arms the index and populates
  `_discTopics`.
- The one-publish-per-tick admission policy from 2.6.0. Each entity
  still publishes on its own main-loop iteration.

## Risks

1. **Silent argument-order swap** in a row. Mitigation: group-wise
   transcription + HA verification per group. (See Step C above.)
2. **Anonymous union confusion** if I decide to compress the entry
   struct further. Stick with the flat struct — clarity beats LOC
   savings.
3. **PROGMEM placement.** A 57-row array of pointers + scalars is
   probably ~5 KB in `.rodata`. Fits easily in flash; no special
   `PROGMEM` annotation needed for ESP32 (unlike AVR — ESP32 puts
   `static constexpr` in flash by default).
4. **Adding a new entity** later requires choosing a row index
   that doesn't shift existing indices (HA's discovery is keyed
   by `object_id`, not index, so reordering rows is safe — but
   the *count* matters for the "are we done discovering" check).

## Out of scope (separate tasks)

- Removing the no-op slot at index 9. That's a separate decision
  about whether to renumber and what discovery deletes to publish.
- Compiling the entire `value_template` strings into a smaller
  shared lookup. Lots of them are unique enough that it's not
  obviously a win.
- Moving discovery topic strings out of `_discTopics` runtime
  fields into compile-time constants. Doable but unrelated.
