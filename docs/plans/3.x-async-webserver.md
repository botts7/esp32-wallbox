# Research: AsyncWebServer migration for the gateway web layer

Target: **3.x** (task #78). Research only — no implementation
commitment.

## What we're considering

Replace the sync Arduino `WebServer` (40 routes in `src/wb_web.cpp`)
with an event-driven async server, so request handlers run on a
dedicated FreeRTOS task instead of on the main loop. The known
candidate library is the
[`ESPAsyncWebServer`](https://github.com/mathieucarbou/ESPAsyncWebServer)
fork — me-no-dev's original is archived, mathieucarbou's fork is
actively maintained and what ESPHome uses (hence the roadmap entry
calling it the "ESPHome AsyncWebServer backport").

## Why we'd want this

2.7.0 cut the worst symptom (HA toggles no longer block the main
task) but the web `/api/command` path still occupies the main task
for up to `?wait` (default 5 s, clamp 8 s) on a slow BAPI call. The
chunked-wait pump (step 9c) keeps MQTT + WS responsive during that
window, but the main loop's own iteration time grows by the same
amount. That's the "known constraint" called out in the 2.7.0
CHANGELOG.

Going async would eliminate the main-loop block entirely:

| | Today (sync) | After async |
|---|---|---|
| Web request blocks main task | Yes, up to `?wait` | No — handler on AsyncTCP task |
| `loop_max_ms` typical under web load | 500-5000 ms | < 50 ms |
| MQTT publish during BAPI wait | Chunked pump keeps it alive | Native — different task |
| WS push during BAPI wait | Chunked pump keeps it alive | Native — different task |
| Re-entrancy class of bug | Tracked via `g_webMaxReentry` | Architecturally impossible |

The 2.7.0 step 9c "chunked-wait pump" was effectively emulating
what async gives natively. Migrating eliminates the pump (and
its 50 ms ticks of MQTT/WS latency).

## Current sync surface (audit)

- **40 routes** registered via `http.on(...)` in
  `src/wb_web.cpp`. Plus 4 catch-alls (`onNotFound`, captive-portal
  redirect, manifest, sw.js).
- **18 distinct `http.*` methods** used: `on / onNotFound / begin /
  handleClient / send / send_P / sendContent / sendHeader /
  setContentLength / arg / hasArg / hasHeader / header /
  collectHeaders / requestAuthentication / authenticate / upload /
  uri`. All have AsyncWebServer equivalents but the signatures
  differ.

Routes break down into roughly six categories:

| Category | Count | Difficulty | Notes |
|---|---|---|---|
| Read-only HTML pages | 6 | Easy | Dashboard, /settings (now JS-injects from /settings/body.gz), /config, /info, /logs, /sessions |
| Read-only JSON endpoints | ~14 | Easy | /api/status, /api/charger, /api/health, /api/diag/*, /api/logs, /api/boot/history, /api/ota/history, /api/config/export, /api/health endpoints |
| State-mutating POSTs | ~7 | Medium | /save, /reset, /api/reboot, /api/pin, /api/diag/clear, /api/fw/dismiss, /api/config/import |
| BLE-passthrough | 3 | Easy (post-2.7.0!) | /api/command, /api/command_status, /api/ble-scan — already async-safe via the BLE request queue from task #71 |
| OTA upload | 1 | **High** | /api/ota — file streaming with different handler signature. **Highest brick-risk route.** |
| Long chunked HTML | 2 | Medium | /settings stub + /info — both currently use `setContentLength(CONTENT_LENGTH_UNKNOWN)` + multiple `sendContent` calls. AsyncWebServer has `beginChunkedResponse` with a fill-callback. |

## What changes

### Handler signature

```cpp
// Today (sync):
http.on("/api/health", []() {
    if (!checkAuth()) return;
    http.send(200, "application/json", buildHealthJson());
});

// After (async):
server.on("/api/health", HTTP_GET, [](AsyncWebServerRequest* req) {
    if (!_checkAuthAsync(req)) return;
    req->send(200, "application/json", buildHealthJson());
});
```

Roughly 1:1 for simple endpoints. The signature change is mechanical.

### Request/response object lifetime

This is the big mental model shift. The sync model is:

> Handler runs, builds response, calls `http.send`, returns.
> Everything happens on the call stack of `handleClient()`.

The async model is:

> Handler receives an `AsyncWebServerRequest*`. It can either send
> a response immediately (synchronously builds + queues for write)
> OR defer the send by holding the pointer and calling `req->send`
> later from another task. **If the handler returns without
> sending, the request is leaked** and the client times out.

For most of our routes this is fine — we send immediately. For
`/api/command` though, the natural deferred pattern is:

```cpp
server.on("/api/command", HTTP_GET, [](AsyncWebServerRequest* req) {
    // ... parse args ...
    uint32_t reqId = wallboxBLE.enqueueRequest(met, par, ReplyMode::ASYNC_WAKE, ...);
    // Stash req pointer keyed by reqId
    _pendingAsyncWebRequests[reqId] = req;
    // No req->send() here — the BLE task wakes us via callback
});

// In the BLE task's drain block:
void _onBleResponse(uint32_t reqId, const String& json) {
    if (auto it = _pendingAsyncWebRequests.find(reqId); ...) {
        // CAUTION: req->send must run on the AsyncTCP task, not BLE
        // task. Marshal back via xQueueSend → AsyncTCP task drains.
        // ESPAsyncWebServer doesn't have native cross-task req->send
        // safety, so this needs a small adapter.
    }
}
```

This is more work than today's chunked-pump approach but gives us
true zero-main-loop-block.

### File upload signature

The `/api/ota` route uses `http.upload()`. AsyncWebServer's upload
handler signature is completely different — receives chunks via a
callback signature with `(req, filename, index, data, len, final)`.
Rewriting handleOtaUpload is the **highest brick-risk** part of
the migration.

### Captive portal

Current code uses `DNSServer` alongside `WebServer`. AsyncWebServer
has its own `AsyncDNSServer` in some forks; mathieucarbou's main
fork plays nicely with the stock `DNSServer` provided the captive-
portal redirect uses `req->redirect(...)` instead of the sync
`http.sendHeader("Location",...)` pattern. Test required.

### Auth

`http.authenticate(user, pass)` becomes
`req->authenticate(user, pass)`. CSRF check needs to read the
header off `req->getHeader("X-CSRF-Token")` instead of `http.header`.
Rate-limit token bucket logic carries over unchanged (lives in
file-static state, not on the request).

## Risks

1. **Library churn.** me-no-dev's archived original. mathieucarbou's
   fork has had breaking changes between major versions. **Pin a
   known-good tag** in `platformio.ini` (`lib_deps =
   mathieucarbou/ESPAsyncWebServer @ X.Y.Z`). Do NOT use `@^X`
   wildcards — minor-version breakage has happened.

2. **PubSubClient + AsyncTCP coexistence.** Both use the LWIP
   socket layer. Reports of LWIP socket exhaustion under load
   when sync and async TCP coexist. Need to test under the
   hardening burst harness. Mitigation: PubSubClient's socket
   timeout is already 1 s (set in step 9c-era 2.6.0 work).

3. **Memory pressure.** AsyncWebServer + AsyncTCP need a baseline
   of ~30-50 KB more RAM (TCP buffers + request queue). Our
   typical free heap is ~130 KB. Headroom is tight; need to
   measure after migration. The longevity.py memory soak test
   directly tells us if it's a problem.

4. **OTA brick risk.** The `/api/ota` upload handler is the route
   that most needs to NOT regress. If the async upload signature
   is wired wrong, a partial write into the OTA partition can
   leave a board needing serial recovery. Mitigation: convert
   OTA route LAST, after every other route is async and known
   working. Keep a known-good sync-OTA binary on hand to flash
   over USB if needed.

5. **Re-entrancy:** AsyncWebServer doesn't have the
   "handleClient inside handleClient" reentrancy bug we currently
   track via `g_webReentryDepth` / `g_webMaxReentry`. The
   reentrancy class of bug becomes architecturally impossible.
   That's a win — but it also means we can drop the tripwire
   plumbing post-migration.

6. **Service worker / PWA.** /sw.js, /manifest.json — both static
   responses. Easy migration but verify the response headers HA
   uses still match.

7. **The 2.8.0 /settings/body.gz win still applies.** Build-time
   gzip + PROGMEM blob works exactly the same on async — only the
   route handler signature changes.

## Migration strategy options

### A) All-at-once swap

Branch off, rewrite all 40 routes, swap library, hope for the
best. Quick if it works; expensive to debug if it doesn't.

### B) Port coexistence

Run async on a different port (e.g., 8080) during migration.
Per route, mirror onto both. Once a route is verified on 8080,
remove the port-80 sync version. Once all routes are on 8080,
swap the listen port. Two TCP servers running at once burns
RAM during the migration but allows incremental verification.

### C) Route-by-route flag

Same pattern as task #77. Add `WB_ASYNC_WEB=0` build flag.
Each migrated route is gated:

```cpp
#if WB_ASYNC_WEB
server.on("/api/health", HTTP_GET, _asyncHandleHealth);
#else
http.on("/api/health", _syncHandleHealth);
#endif
```

When the last route migrates, flip the default to 1; next release
drops the sync paths. **This is what I'd pick.** Compatible with
how #77 ships, easy mental model, every commit builds clean both
ways.

## Decision points to bring to the maintainer

Before any code work starts:

1. **Library choice.** mathieucarbou/ESPAsyncWebServer (active fork,
   ESPHome uses it) vs the original me-no-dev (archived). My
   recommendation: mathieucarbou, pinned to a specific tagged
   release.
2. **Memory cost acceptance.** +30-50 KB RAM is real. Are we OK
   with that, or do we need to find compensating savings first
   (e.g., shrink the BLE NimBLE config)?
3. **Migration strategy.** A (all-at-once), B (port coexistence),
   or C (route-by-route flag). Defaulting to C per the
   recommendation above.
4. **Ship vehicle.** Is this 3.0 (alongside the table-driven
   discovery refactor #77) or its own release 3.1.0? My
   recommendation: SEPARATE release. Bundling 3.0 + this in one
   release maximises blast radius if anything regresses; either
   refactor alone is one release worth of testing.
5. **OTA fallback.** Should we keep a sync-OTA escape hatch in
   3.x firmware until async-OTA is field-tested? E.g., the
   `/api/ota` route stays sync (mounted on the legacy WebServer
   on port 81) while every other route is on async port 80, until
   we trust async-OTA. Reduces brick risk.
