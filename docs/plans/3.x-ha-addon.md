# Research: Home Assistant Add-on packaging

Target: **3.x** (task #79). Research only — no implementation
commitment.

## What we're considering

Package this project as a Home Assistant **Add-on** so users on
HAOS / Supervised installs can install the gateway management UI
from HA's Add-on store instead of (or in addition to) the current
"download firmware from the GitHub Releases page → flash via
esp-web-tools" path.

Two important distinctions up front:

| | Add-on | Custom Integration (HACS) |
|---|---|---|
| What runs | A Docker container on the HA host (HAOS / Supervised) | Python custom_component inside HA core |
| Distribution | A GitHub repo URL the user adds to HA's Add-on Store | Either core HA, HACS, or a `custom_components/` drop |
| Talks to | Anything on the network (or USB if mapped) | HA's bus, services, entities |
| Good for | Standalone services, gateways, UI tools | Per-entity wiring of an external device |

Today the gateway publishes MQTT discovery and HA picks up the
entities automatically via the existing MQTT integration. That's
fine for entities. What's missing is a **gateway-management UI
inside HA** — there's currently no HA-native way to see the
gateway's BLE state, recent loop_max_ms events, or fire an OTA
without leaving HA.

The Add-on would provide that UI.

This task is specifically about the **Add-on** path (per the
roadmap entry). A HACS integration is a separate, complementary
option discussed in "Out of scope" at the bottom.

## What the Add-on would do

User opens HA → Settings → Add-ons → "Wallbox BLE Gateway" → Open
Web UI. They see:

- **Status panel:** gateway IP, BLE state, charger name, current
  charging power, last reconnect counters. Sourced from
  `/api/health` + `/api/diag/disconnects` + `/api/charger` on the
  ESP32.
- **OTA section:** drag-and-drop firmware.bin upload that posts
  to the gateway's `/api/ota` endpoint. Optionally auto-fetches
  the latest release from the GitHub Releases API and offers a
  one-click upgrade with rollback.
- **Diagnostics card:** mirrors `/info` → Connection Diagnostics
  but in a HA-native styled card. Smart tripwire ring (2.8.0
  task #74) renders nicely here.
- **Config link:** deep-links to the gateway's `/config` page in
  a new tab (the Add-on doesn't try to re-host the config UI;
  the gateway's own captive-portal-style config is fine).
- **Discovery:** uses HA's network to mDNS-resolve
  `wallbox-gw.local`, or scans a small range, or accepts a
  manually-entered IP. Persists the IP in the Add-on's own state.

It does NOT replace anything the ESP32 does — it's a thin
forward proxy + dashboard. The actual BAPI talking, MQTT
publishing, and BLE link stay on the ESP32.

## Add-on anatomy

An HA Add-on is essentially a Docker image plus metadata. The
minimum viable layout for our case:

```
ha-addon-wallbox/                 # separate repo OR subdir
├── README.md                     # what the Add-on does
├── repository.yaml               # tells HA this folder is an Add-on repo
└── wallbox-gateway/              # the Add-on itself
    ├── config.yaml               # name, version, options, ports, etc.
    ├── Dockerfile                # builds the runtime image
    ├── rootfs/                   # files copied into the image
    │   └── etc/services.d/
    │       └── wallbox/
    │           └── run           # s6-overlay entrypoint
    ├── app/                      # the actual Python app
    │   ├── server.py             # Flask service (see "Implementation choice")
    │   ├── proxy.py              # /api/* passthrough to gateway IP
    │   ├── static/
    │   └── templates/
    ├── icon.png                  # 256x256
    ├── logo.png                  # 250x100
    └── translations/
        └── en.yaml
```

The Add-on `config.yaml` describes how Supervisor runs it:

```yaml
name: "Wallbox BLE Gateway Manager"
version: "0.1.0"
slug: wallbox_gateway
description: |
  Manage your ESP32-based Wallbox BLE gateway from inside HA.
  Status, diagnostics, OTA firmware updates.
arch:
  - amd64
  - aarch64
  - armhf
  - armv7
  - i386
init: false        # use s6-overlay (HA's standard init)
ingress: true      # HA proxies the Add-on's port behind /api/hassio_ingress/
ingress_port: 8099
panel_icon: mdi:ev-station
panel_title: "Wallbox Gateway"
homeassistant_api: false   # we don't need to talk to HA core
hassio_api: false          # we don't need to talk to Supervisor
options:
  gateway_ip: ""
  poll_interval_s: 30
schema:
  gateway_ip: str?
  poll_interval_s: int(5,300)?
```

The `ingress` field is the magic that makes the Add-on appear in
HA's sidebar with a panel icon. HA handles auth + routing
transparently; the Add-on just sees plain HTTP on port 8099.

## Implementation choice: language

| Option | Pros | Cons |
|---|---|---|
| **Python + Flask** | Mature, narrow attack surface, no Starlette dependency, easy to debug | Synchronous — but our workload is request/response with no streaming needs |
| Python + FastAPI | Async, OpenAPI, Pydantic | Starlette dependency: see CVE block below. Plus more deps to keep patched |
| Node.js + Express | Smaller image, faster start | Less common in HA Add-on ecosystem; less reference code |
| Bash + busybox + caddy-as-proxy | Tiny image | UI work much harder, can't reuse Python OTA libraries |
| Static SPA + nginx + tiny proxy | Zero runtime backend logic | Still need *something* to proxy OTA POSTs and add headers |

**Recommendation: Python + Flask.** Matches HA ecosystem
conventions (Supervisor itself is Python, most community Add-ons
are Python), but Flask gives us a smaller dependency tree than
FastAPI and avoids the Starlette CVE churn discussed below. Our
workload is plain request/response with no async benefit:
~5 endpoints, one upstream HTTP call per request, plus one file
upload proxy. Flask covers all of that in ~100 LOC.

### Note on FastAPI security posture (2026)

Mid-2026 saw multiple high-severity CVEs in FastAPI and its
Starlette dependency:

- **CVE-2026-48710 (BadHost)** — Starlette host-header validation
  bypass; unauthenticated attacker can slip past access controls
  via a malformed Host header. Fix: Starlette ≥ 1.0.1.
- **CVE-2026-2978** — Critical RCE in FastAPI < 0.115.8.
  CVSS 9.8. Fix: FastAPI ≥ 0.115.8.
- **CVE-2026-23996** — API-key timing attack in the FastAPI
  ApiKey library. Doesn't affect us (we don't use that auth
  pattern — HA ingress handles user auth) but illustrates the
  pace of issues in the ecosystem.

Our Add-on context mitigates these architecturally — we run
behind HA's Supervisor ingress proxy, never directly internet-
exposed, and auth is HA's responsibility — but the takeaway is
that the FastAPI dependency tree adds review burden we don't
need for our actual workload. Flask has had fewer
high-severity CVEs in the same window and a smaller surface
to patch.

If a future scope expansion needs async (e.g. streaming logs
from many gateways concurrently), revisit FastAPI with pinned-
and-monitored versions at that point.

## Functional scope phases

A defensible 0.x → 1.0 progression:

### v0.1 — Discover + Read-only dashboard

- mDNS discovery of `wallbox-gw.local`, fallback to user-entered IP
- Read-only mirror of `/api/health`, `/api/diag/disconnects`,
  `/api/charger`
- HA-native styled status card with live updates (poll every 10s)
- No state mutation. Safe to ship — can't break anything.

### v0.2 — OTA upload

- Drag-and-drop firmware.bin upload
- Forwards as `POST /api/ota` to the gateway with the gateway's
  OTA password (entered once in Add-on config)
- Progress bar; success/failure reported back to the UI
- "Latest release from GitHub" auto-fetch + one-click upgrade
  (optional, behind a toggle)

### v0.3 — Polish

- Show recent loop_max_ms events (2.8.0 task #74's ring) in a
  sparkline
- Show NVS-persisted disconnect history from prior boots
- Link to /config in a new tab (no in-Add-on config editor —
  gateway's own UI is fine and we don't want to duplicate it)

### v1.0 — Promote to official repository

If usage justifies it: PR to HA's community Add-on repository.
Many gateways like this live in `hassio-addons/repository` — a
collaborative maintained set of community Add-ons.

## Repository layout decision

| Approach | Pros | Cons |
|---|---|---|
| **Subdir in this repo** (`ha-addon/`) | One place to update; firmware + Add-on versions ship together | Anyone adding the repo to HA's store ALSO gets the firmware source they don't need; uglier UX |
| **Separate repo** (`botts7/wallbox-gateway-ha-addon`) | Clean Add-on repo URL for HA users; tagged independently | Two repos to maintain; firmware + Add-on version drift is possible |

Recommendation: **separate repo.** HA Add-on store URLs are
user-pasted; they should be clean. Firmware/Add-on versions
already drift fine (Add-on talks to firmware over a stable HTTP
API; the contract is the API, not co-versioning).

## Risks

1. **Add-on review for official store.** If we PR to
   community-hassio-addons, the maintainers may push back on
   architecture, security posture, multi-arch builds. Lots of
   work for marginal distribution gain unless we have many
   users on HAOS. **Recommendation: ship as a custom repo
   first; PR upstream only after real adoption.**

2. **Gateway auth credentials.** The Add-on needs the gateway's
   web-auth + OTA password. Storing it in Add-on options is fine
   (HA encrypts options at rest) but document the cred-rotation
   story clearly — if a user changes the gateway password they
   have to update the Add-on options.

3. **HTTPS gateway-side.** ESP32 doesn't do HTTPS; Add-on
   traffic to the gateway is plain HTTP. Inside an HAOS network
   that's fine, but call it out in the Add-on description for
   users who route HA traffic externally.

4. **Multi-arch Docker builds.** Five architectures (amd64,
   aarch64, armhf, armv7, i386). GitHub Actions can build all
   five via QEMU but adds build time. For v0.1 we can target
   amd64 + aarch64 only (covers most HAOS deployments).

5. **OTA file size.** Our firmware is ~1.3 MB. POST size limits
   in the Add-on's web framework + the gateway's chunked OTA
   handler both need to accommodate. We already handle 1.3 MB
   uploads via the existing /ota web page — the Add-on just
   proxies that.

6. **mDNS inside HA's network.** HAOS network namespaces can
   sometimes break mDNS resolution. Fallback to user-entered
   IP must always be available.

7. **Ingress + session state.** HA's ingress proxy strips
   cookies. If the Add-on stores session state, use Flask's
   server-side session (filesystem or sqlite) keyed by the
   ingress request's X-Ingress-Path / X-Forwarded-* headers.
   For v0.1-v0.3 we don't store session state at all — the
   gateway IP lives in Add-on options, dashboard data is
   live-polled.

## What we get / what we don't

What this DOES improve:
- HA-native UI for the gateway. Users who live in HA's UI all day
  get gateway management without leaving HA.
- One-click "update to latest" OTA flow.
- Discoverable distribution: HA's Add-on store has higher
  visibility than a GitHub Releases page for some users.

What this does NOT change:
- The gateway's BLE → MQTT bridge still does its own thing on
  the ESP32.
- HA still picks up entities via the existing MQTT discovery.
- Users on non-HAOS HA installations (Docker, venv) can't run
  Add-ons; the Add-on is only useful for HAOS / Supervised.

## Decision points to bring to the maintainer

Before code work starts:

1. **Repo layout:** subdir of this repo, or new
   `wallbox-gateway-ha-addon` repo? My recommendation: separate
   repo.
2. **Language / framework:** Python + Flask (revised
   recommendation, was FastAPI in v1 of this doc — see the
   security-posture block above for the reason).
3. **Scope cut for v0.1:** read-only dashboard only, or include
   OTA from day one? My recommendation: read-only v0.1, OTA in
   v0.2 — earlier sanity feedback before touching the brick-risk
   OTA path.
4. **Distribution path:** custom repo URL only (forever), or
   target the community-hassio-addons official repo
   eventually? My recommendation: custom repo first, official
   PR only after real usage.
5. **Multi-arch:** v0.1 amd64 + aarch64 only, or all five
   architectures from day one? My recommendation: 2 archs for
   v0.1.

## Out of scope (separate possible tasks)

1. **HACS custom integration.** A HACS-distributed
   `wallbox_gateway` custom_component that exposes the gateway's
   diagnostics as native HA sensors instead of via the MQTT
   bridge. Different audience (Docker / venv HA installs, not
   just HAOS). Could happen in parallel with the Add-on or
   instead of it. Larger Python codebase since it integrates
   with HA core directly.
2. **Migration from MQTT-discovery to native integration.** If
   we ever ship the HACS integration, the gateway would no
   longer need to publish HA discovery payloads — they'd come
   from the integration directly. That's a 4.x conversation,
   not 3.x.
3. **Multi-gateway support in the Add-on.** v1.x stretch — the
   Add-on UI managing multiple gateways at once for users with
   several chargers. Maps cleanly onto the existing single-IP
   approach (just a list of IPs and tabs).
